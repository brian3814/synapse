TL;DR — How Synapse Works
1
Two processes, one bridge. Electron main process owns SQLite, LLM calls, filesystem, and embeddings. Renderer is a React SPA. They communicate through 45+ IPC channels via a preload contextBridge.
2
Vault = everything in one folder. A vault directory holds graph.db, notes/, embeddings, and user files. One vault per process. Switching vaults spawns a new OS process.
3
Platform abstraction via build alias. UI imports @platform — resolved at build time to Chrome or Electron implementations. 7 interfaces (Storage, DB, Notes, LLM, Browser, Embedding, Vault) make UI code platform-agnostic.
4
Event-driven vault sync. FileWatcher → EventBus → handlers (NoteFile, ResourceDetection, SyncBroadcast). Handlers subscribe independently; SyncBroadcastHandler bridges vault events to IPC for the renderer.
5
DB is engine-swappable. UI calls db.request(action, params). ActionHandler (96 cases) dispatches to DataStore (16 repo sub-interfaces). Swap SQLite for Postgres by implementing one interface.
6
LLM calls are provider-neutral. Core llm-protocol.ts defines StreamFn + LLMMessage. electron/llm-backend.ts routes via provider factory (getStreamFn(provider)). Anthropic registered; OpenAI/others plug in via registerStreamFn(). Two tool systems: 9 extraction tools and 17 chat tools.
7
Three-layer graph rendering. Three.js InstancedMesh (WebGL) for nodes/edges, Canvas2D overlay for labels, React DOM for panels. Layout runs in a Web Worker (Barnes-Hut force-directed). 10 Zustand stores, no cross-store mutations.
8
Companion extension for page capture. Separate Chrome extension POSTs rendered DOM to localhost:19876. Main process broadcasts via IPC. Unidirectional: browser → desktop only.

Critical Dataflows

The five fundamental data paths through the system. Every user-facing feature maps to one or more of these flows. The diagram below shows all five simultaneously — scroll horizontally on smaller screens.

Flow Summaries

1. Page Capture
Companion extension captures rendered DOM as markdown → HTTP POST to localhost:19876 → CompanionServer broadcasts companion:capture via IPC → useCompanionCapture hook sets pending capture in LLM store → opens extraction modal.
2. LLM Extraction
useLLMExtraction invokes llm:stream-extraction (fire-and-forget) → Main process routes to provider via getStreamFn(provider) factory → SSE stream → chunks broadcast back via llm:extraction-chunk matched by requestId → Zod parse → entity resolution → ExtractionReviewTab with undo/redo → applyReview() commits to DB.
3. DB + Graph Sync
db-client invokes db:request → ActionHandler dispatches to DataStore (16 repos) → SQLite mutation → SyncEvent emitted → broadcast db:sync to all windows → graph-store.startSyncListener() updates nodes/edges/adjacency → GraphRenderer re-renders. Also triggers vault EventBus for file-level handlers.
4. File Detection
User drops file into vault directory → fs.watch fires (debounced 500ms) → FileWatcher emits file:added on EventBus → ResourceDetectionHandler checks sandbox rules → creates resource node in DB → emits node:created → cascades into Flow 3 (DB sync) and Flow 5 (embedding).
5. Auto-Embedding
DB mutation emits node_created/node_updated → EmbeddingService.handleNodeMutation() → builds embedding text (strategy per node type) → computes hash (skips if unchanged) → enqueues to EmbeddingQueue → provider embeds (ONNX local or OpenAI API) → stores vector in sqlite-vec vec_nodes table.

Bridges internal vault events to renderer via IPC. Subscribes to all node:* and edge:* events, broadcasts on vault:sync channel to all BrowserWindow instances.

Agent Loop (src/core/agent-loop.ts)

The loop accepts three injectable dependencies: streamFn (LLM streaming impl), toolExecutor (platform-specific tool dispatch), and onProgress (event callback). This keeps the core loop platform-agnostic.

1
Filter Tools & Build Prompt
Remove disabled tools (from AgentToolConfig), but always keep save_entities. Build system prompt with optional custom instructions. Init messages: [user prompt].
2
LLM Call
streamFn() → streamAnthropicWithTools() → SSE streaming → parse text content + tool_use blocks. Emit llm_start, llm_chunk, llm_end events.
3
Check Terminal Tool
If any tool call is save_entities → parse input as ExtractionResult, emit extraction_complete, exit loop immediately. The tool is never executed — its input IS the extraction output.
4
Execute Tools Sequentially
For each ToolCall: toolExecutor.execute(tc) → returns {result, error?}. Emit tool_call and tool_result events. Error results get is_error: true flag.
5
Build Anthropic Messages
Push assistant message (text + tool_use blocks), then user message (all tool_result blocks in a single message). This is the Anthropic API multi-turn tool-use format.
6
Loop
Back to step 2 unless maxIterations (default 15) reached or terminal tool was called.

Anthropic Message Format

Both extraction and chat loops build message arrays using Anthropic's multi-turn tool-use protocol. The content block union type carries text, tool invocations, and tool results in a single conversation thread.

text
{ type: 'text', text: string }
LLM's natural language reasoning. Appears in assistant messages.
tool_use
{ type: 'tool_use', id, name, input }
LLM requesting tool execution. ID generated by API for correlation.
tool_result
{ type: 'tool_result', tool_use_id, content, is_error? }
Tool output sent back in next user message. Multiple results bundled in one message.

Filtering Rules

  • Extraction: save_entities is always included regardless of disabled list (it's the only way to complete extraction).
  • Chat: All 16 base tools filterable. semantic_search independently gated by platform (platformId === 'electron') AND disabled list.
  • Timing: Filtering happens before converting to Anthropic format — disabled tools are never sent to the LLM, so the model doesn't even know they exist.

Design Principles

  • Dependency injection over inheritance: The core agent loop accepts streamFn, toolExecutor, and onProgress as arguments. No abstract base classes, no platform-specific imports. Adding a new platform means writing one ToolExecutor implementation.
  • Terminal tool pattern: save_entities is never executed — its input field IS the extraction output. This lets the LLM decide when extraction is complete without needing a separate "done" signal.
  • Execution context tags: Each tool definition carries executionContext: 'content-script' | 'offscreen' | 'ui'. The ToolExecutor implementation uses this tag to route dispatch. Adding a new context (e.g., 'worker') requires only a new tag value and executor branch.
  • Dynamic tool registration: semantic_search is appended at runtime based on platform capability. This pattern extends naturally to any future platform-gated tool.
  • Subgraph accumulation: Chat tools return collected entity IDs alongside their text results. The agent loop accumulates these across iterations, producing a referenced subgraph in the final event. This separates data collection from visualization without coupling tools to the rendering layer.
  • Filtering is pre-LLM: Disabled tools are removed before sending the tool list to the Anthropic API. The model never sees disabled tools, preventing hallucinated calls to unavailable tools.

Key Design Principles

  • Graph-as-registry: SQLite is the source of truth. Filesystem is a projection. Every file with a graph node has vault_path set on the node.
  • Event-driven: Graph mutations and file events flow through VaultEventBus. Handlers subscribe independently.
  • Platform-agnostic UI: Same React code runs on both Electron and Chrome via @platform alias. Zero platform-specific imports in UI components.
  • Shared DB handle: VaultManager.open() calls resetBetterSQLite(dbPath) then runMigrations(). All code shares one DB handle.
  • Lazy initialization: EmbeddingService initializes after first DB init, not during app.whenReady(). Avoids race conditions.
  • Streaming-first LLM: All LLM operations stream tokens in real-time. Fire-and-forget invoke, responses matched by requestId.
  • Local-first: All data stays in the vault directory. API keys stored in app settings, never in vault.
